# BatchImager Product Overview

## 产品定位

BatchImager 是一个面向商品图生产的桌面工作台。用户把库房、仓库或现场拍摄的原始商品图片导入到工作区，再用批量提示词或单图会话提示词生成更适合电商展示的商品图。

第一阶段先完成真实产品框架：跨 Windows/macOS 的 Electron 应用、图片导入、拖拽导入、可配置列数的图片工作区、右侧会话区骨架，以及可复用的图生图 API 调用层。

## 核心工作流

1. 用户通过“导入”从本地多选图片，或者直接把图片拖拽到工作区。
2. 应用为每张图片创建一个图像会话，并在左侧工作区按 N 列展示，默认 4 列。
3. 图片之间只用细线分隔，文件名和状态图标融入图片内部，尽量保留图片内容面积。
4. 点击图片后，右侧约 1/4 宽度的会话区显示该图片的会话上下文。
5. 用户可通过“批量处理”弹窗输入本轮提示词并并行触发多图生成，也可在单图会话区微调某一张图片。
6. 生成完成后，应用把远端返回图片下载到本地用户数据目录，并用新图片替换工作区中的显示图。

## 第一阶段范围

- 支持 Windows 和 macOS 的 Electron 桌面应用框架。
- 支持从系统文件选择器多选图片导入。
- 支持拖拽图片文件到工作区导入。
- 过滤非图片文件，避免重复导入同一路径。
- 用真实图片文件路径展示缩略图。
- 提供可配置列数，默认 4 列。
- 提供右侧会话区骨架，展示选中图片的信息、最近提示词和生成状态。
- 接入 Tuzi/OpenAI Images 兼容的图生图接口，支持批量生成和单图微调。
- 图像生成 API key 存放在本地 `.env.local`，只由 Electron main 进程读取。
- 图像生成前会透明地把输入图转换为 PNG，并在不裁剪、不拉伸的前提下等比压缩到接口允许范围；默认输出尺寸使用处理后的输入图尺寸，只有用户明确要求比例或尺寸时才覆盖。
- 提供“日志”入口，实时展示用户可读的运行进度；后台同时输出更详细的控制台和本地日志。

## 暂不包含

- 图片生成版本管理。
- 本地项目文件保存和恢复。
- 导出、批量下载、云同步、账号体系。
- 复杂素材库、模板市场和团队协作。

## 技术选型

- Electron：跨 Windows/macOS 的桌面外壳、文件选择器、系统能力。
- Vite：快速开发服务器和前端构建。
- React + TypeScript：构建状态清晰的桌面工作区 UI。
- Vitest：测试纯业务逻辑，如图片过滤、去重、会话创建。
- CSS Modules/普通 CSS：第一阶段直接使用局部清晰的 CSS，避免过早引入组件库。
- Tuzi 图像 API：通过 OpenAI Images 兼容的 `POST /v1/images/edits` 实现图生图；调用集中在 Electron main 进程，renderer 只通过 IPC 发起任务。
- Sharp：在 Electron main 进程完成本地图像入参标准化，包括格式转换、EXIF 自动旋转和等比压缩。

## 架构边界

- `electron/main`：应用生命周期、窗口创建、文件选择器、IPC 处理。
- `electron/services`：本地配置读取、图生图 API client、结果图片落盘。
- `electron/services/imageEditInput`：图像 edit API 的输入预处理，保持原图比例，最长边默认不超过 3840，PNG 文件控制在 4MB 内。
- `electron/services/appLogger`：后台详细日志、控制台输出和前台实时日志事件。
- `electron/preload`：以最小安全桥接方式暴露渲染层可用 API。
- `src/domain`：与 UI 无关的图像会话和文件导入逻辑。
- `src/ui`：React 组件、交互和视觉层。
- `src/types`：跨主进程、preload、渲染层共享的类型。

## 设计原则

- 图片优先：网格尽量少用容器和装饰，图片之间用 1px 细线分割。
- 中文优先：应用界面文字默认中文。
- 桌面密度：按 macOS/Windows 桌面软件尺度设计，不使用过大的网页式控件。
- 渐进实现：第一阶段把导入和展示做稳，生成能力通过接口接入，不做假功能。
- 安全边界：渲染层不直接访问 Node 文件系统，只通过 preload 调用受控能力。
